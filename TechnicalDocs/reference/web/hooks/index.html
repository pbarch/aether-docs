Hooks & Signals¶

Signals are used to automatically update UI components when the state of the application changes. Signals in Aether's UI make use of Preact's Hooks and Signals, as well as Aether's Observable object to simplify creating UI components that are sync'd with Aether's Database.

Each type of Signal references a Observable object and a key which defines the value on the State Proxy object that is being listened to, or changed.

When to Use Signals

Signals should be used in contexts where the current state of the application needs to be kept in sync with the database, but there is no re-occurring update loop. For contexts with an update loop, like a p5.js canvas, us a get() (or one of the get helpers) instead.

When to Use use

Each signal described below has a memoized varient use<SingalName>. These memoized hooks can only be used within Preact function components. When creating a signal outside of a Preact component, use the standard version of the signal not the use<SingalName>, version.

objectSignal():¶

objectSignal takes a Preact Signal, a Observable object and a Key. When the value at the specified Key is changed (say by a UI component using an objectSignal) the signal updates the State Proxy object, which then updates the database. If the Key that the objectSignal is listening to is updated in the database the signal value will be updated as well.

The useObjetSignal function takes only the Observable object and Key, and creates a memoized objectSignal.

errorSignal():¶

errorSignal takes a Preact Signal, a Observable object, and a Key. Like objectSignal, when the value at the specified Key is changed (say by a UI component using an the errorSignal) the signal updates the State Proxy object, which then updates the database. If the Key that the errorSignal is listening to is updated in the database the signal value will be updated as well.

an errorSignal also keeps track of whether or not its Observable object was successful in updating the database. If updating the database was unsuccessful, then the error signal reverts the Observable's local value back to the last successfully set value.

The useErrorSignal function takes only the Observable object and Key, and creates a memoized errorSignal.

dependentSignal():¶

dependentSignal takes a dependency (another signal), a "getObj" expression (which returns an Observable object, typically using the logic dependent on the value of the dependency), and a Key. dependentSignal can be used to await loading, and to setup signals for values who's database path changes depending on the current state of the application (Influence Engine parameters in live mode or not, for example).

If the dependency is undefined the dependentSignal will be in a loading state and also be undefined until the dependency resolves to a value.

After loading, if the value of the dependency changes the dependentSignal recomputes the getObj expression to return a Observable object based on the dependencies new value. The recomputed value of the dependentSignal will then be the Key's corresponding value in the returned Observable object.

The useDependentSignal function creates a memoized dependentSignal.

Example

An example of a dependent signal for the current preset name for cloudGen:

The dependentSignal would be written as:

const currentPresetName = useDependentSignal(live, () => currentSetting("cloudGen"), "name");

Where:

  • live is another signal connected to value at users.<SESSION_ID>.live

  • () => currentSetting("CloudGen") is the getObj expression. currentSetting() is a helper function which returns the object at the module name provided, in this case, "CloudGen". The logic in the currentSetting() function depends on the value at users.<SESSION_ID>.live to determine the path to the object it returns. If live is false currentSetting("CloudGen") returns the object at <TESTBED>.transient.sessions.<SESSION_ID>.CloudGen. If live is true currentSetting("CloudGen") returns the object at <TESTBED>.<SCENE>.CloudGen

  • name is the key within the setting object we want this signal to listen for changes to.

When the value of users.<SESSION_ID>.live changes this dependentSignal is notfied by the live signal, and re-evaluates which Observable object it should listen to the "name" key on, to be either:

<TESTBED>.transient.sessions.<SESSION_ID>.CloudGen> if live is false or,

<TESTBED>.<SCENE>.CloudGen if live is true.

stateSignal

Many dependentSignals use stateSignal as their dependency to wait for the database to be loaded. stateSignal is undefined until the entire database is loaded. Once loading is complete the value of stateSignal contains the state of the entire database.

stateSignal can be imported with:

import {stateSignal} from "../global.js";

liveSignal

The live state of the users session is another common dependency. liveSignal, a signal connected to the live state, can be imported to use as a dependency with: import { liveSignal } from '/libraries/utilities.js'
